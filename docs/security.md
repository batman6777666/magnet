# Security

## API Key Format and Generation

API keys are generated by `POST /auth/register` or by setting `API_KEYS` in the environment.

### Generated Keys

Format: `mk-{timestamp}-{random}-{checksum}`

```
mk-m5z8abc-def1234567890abcdef1234567890-a1b2c3d4
```

- `timestamp`: `Date.now().toString(36)` — base36 millisecond timestamp
- `random`: `crypto.randomBytes(16).toString('hex')` — 32 hex characters of CSPRNG output
- `checksum`: `sha256(timestamp + random).slice(0, 8)` — first 8 hex chars of SHA-256 hash

Generation uses Node.js `crypto.randomBytes`, which draws from the OS entropy source (`/dev/urandom` on Linux, `BCryptGenRandom` on Windows).

### Static Keys

Configured via the `API_KEYS` environment variable as a comma-separated list:

```
API_KEYS=sk_live_abc123,sk_live_def456
```

Static keys bypass the JSON store and are always treated as valid if they appear in the comma-separated list. Note: the generator command shown in `.env` is:

```bash
node -e "console.log('sk_live_' + require('crypto').randomBytes(24).toString('hex'))"
```

### Key Storage

Keys are stored in `backend/data/keys.json`. Each record contains:

```json
{
  "mk-abc...": {
    "name": "User",
    "email": null,
    "active": true,
    "requestCount": 0,
    "createdAt": "...",
    "lastUsedAt": null
  }
}
```

Keys can be revoked by setting `active: false`. The store file must not be committed to version control (it is excluded by `.dockerignore`).

## Rate Limiting

| Endpoint | Scope | Limit | Configuration |
|---|---|---|---|
| `POST /v1/extract` | Per API key | 100 req / 60s | `RATE_LIMIT_WINDOW_MS`, `RATE_LIMIT_MAX` |
| `POST /auth/register` | Per IP | 10 req / hour | Hardcoded in route definition |

Rate limiting headers: `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset`.

The extract rate limiter uses `x-api-key` as the key generator, falling back to `req.ip`. This prevents a single key from abusing the service while allowing legitimate burst usage within the window.

## Helmet Security Headers

Applied globally via `helmet()`. Default protections include:

| Header | Effect |
|---|---|
| `Content-Security-Policy` | Disabled (needed for Puppeteer inline scripts) |
| `X-Content-Type-Options` | `nosniff` — prevents MIME type sniffing |
| `X-Frame-Options` | `SAMEORIGIN` — prevents clickjacking |
| `Strict-Transport-Security` | Enforces HTTPS |
| `X-XSS-Protection` | Enables browser XSS filter |
| `Referrer-Policy` | Controls referrer header leakage |
| `Cross-Origin-Embedder-Policy` | Disabled (needed for cross-origin resources) |

## CORS Configuration

CORS is enabled globally with `cors()`. In production this should be restricted to the frontend domain:

```js
cors({ origin: 'https://your-frontend.pages.dev' })
```

The current default (no options) allows all origins, which simplifies development but should be locked down for production.

## Input Validation

All endpoints validate inputs before processing:

| Endpoint | Validations |
|---|---|
| `POST /auth/register` | `name`: string, 2-80 chars required; `email`: optional, regex-validated format |
| `POST /v1/extract` | `type`: must be `"movie"` or `"series"`; `name`: non-empty string required; `season`/`episode`: positive integers for series |
| `POST /inspect` | `url`: non-empty string, must pass `new URL()` constructor |

Express JSON body parser is limited to `10kb` to prevent large payload attacks.

## No Secrets in Code

- No API keys, passwords, or tokens are hardcoded
- All configuration comes from environment variables or the `.env` file (which is excluded from `.dockerignore` and `.gitignore`)
- `keys.json` is generated at runtime and not committed

## Puppeteer Sandbox Considerations

Puppeteer runs with `--no-sandbox` and `--disable-setuid-sandbox` because Hugging Face Spaces and most Docker environments do not support the Chrome sandbox. This is standard practice for containerized Puppeteer but means the browser process is less isolated from the host.

Additional launch flags for security and stability:

```
--disable-dev-shm-usage
--disable-gpu
--disable-extensions
--no-first-run
--disable-blink-features=AutomationControlled
```

## Abuse Prevention Strategies

1. **Per-key rate limiting**: Stops a single compromised key from exhausting resources
2. **Registration throttling**: 10 keys per hour per IP limits bulk registration
3. **Request timeout**: Hard cap of 30 seconds per extraction prevents slow-loris attacks
4. **Body size limit**: JSON parser limited to 10kb
5. **Static API keys**: Operators can pre-configure known keys and disable dynamic registration
6. **Key revocation**: Keys can be deactivated via the store without redeploying

## Data Storage Considerations

- **No PII stored by default**: The `email` field is optional. No passwords are ever stored.
- **Usage tracking**: `requestCount` and `lastUsedAt` are tracked per key for monitoring but contain no personal data.
- **No session management**: The API is stateless (aside from the key store). No cookies or tokens are issued beyond the API key.
- **Logging**: Console logs include API key name and request paths but never full key values.

## Environment Variable Management

| Variable | Sensitivity | Recommendation |
|---|---|---|
| `API_KEYS` | High | Use secrets manager (HF Space Secrets, GitHub Secrets) |
| `BROWSER_POOL_SIZE` | Low | Commit to `.env.example` |
| `PORT` | Low | Commit to Dockerfile |
| `*_TIMEOUT_MS` | Low | Commit to `.env.example` |
| `RATE_LIMIT_*` | Low | Commit to `.env.example` |

Never commit `.env` files. Use `.env.example` with placeholder values as a template.
